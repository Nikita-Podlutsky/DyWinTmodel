# Техническое руководство DyWinT (ML-Engineering)

Данный документ содержит глубокое описание математики, структуры данных и оптимизаций, реализованных в архитектуре **DyWinT** (Dynamic Windowed Temporal Conformer).

---

## 1. Концепция внутрисетевого окна (In-Model Windowing)

Обычные речевые архитектуры режут аудиофайл на независимые куски до подачи в нейросеть. Это приводит к потере контекста на стыках: если фонема (звук) оказалась разделена границей куска, модель распознает ее неверно.

В **DyWinT** аудио обрабатывается единым потоком:
1.  Сырой сигнал `[B, T_audio]` преобразуется в параллельные фичи спектрограммы (Mel) и коэффициентов (MFCC) формы `[B, n_features, T_spec]`.
2.  Применяется двумерное позиционное кодирование `PositionalEncoding2D` к тензору `[B, T_spec, 1, hid_dim]`. Это сохраняет глобальную временную шкалу для всей аудиозаписи.
3.  Нарезка на перекрывающиеся скользящие окна происходит на латентном уровне внутри модели с помощью операции `unfold` по временной оси:
    ```python
    windows = x_features.unfold(2, self.window_size, self.stride) # Выход: [B, D, N, W]
    ```
    Где `D` — `hid_dim`, `N` — количество окон, `W` — ширина окна. Тензор перестраивается в плоскую форму `[B*N, W, D]` для параллельной обработки Conformer-энкодерами.

---

## 2. Динамический Zero-Padding (Защита от коротких записей)

Если подать в модель аудиозапись короче, чем размер окна `window_size` (например, короткий вздох или буква), операция `unfold` вызовет аппаратное исключение `RuntimeError: maximum size for tensor at dimension 2 is X but size is Y`.

Мы решили эту проблему на уровне препроцессора признаков: если длина спектрограммы `T` оказывается меньше `window_size`, тензор автоматически дополняется нулями справа до минимально допустимого размера:
```python
if T < self.window_size:
    pad_size = self.window_size - T
    x_features = F.pad(x_features, (0, pad_size))  # Дополнение по оси времени
```
Это гарантирует стабильность инференса при любой длине входного сигнала.

## 3. Багфикс Кросс-Внимания (`batch_first=True`)

В исходной реализации была допущена ошибка согласования размерностей в слое `nn.MultiheadAttention`, инициализированном с флагом `batch_first=True`.

На вход слою подавались тензоры, принудительно переставленные через `.permute(1, 0, 2)` в форму `[W, B*N, D]` (Seq, Batch, Dim). Из-за установленного флага `batch_first=True` PyTorch трактовал размерность `W` как размер батча, а `B*N` — как длину последовательности. В результате внимание рассчитывалось **между разными файлами и окнами во всем батче**, вызывая критическую утечку данных.

В текущей версии все лишние перестановки осей удалены, данные поступают в чистом формате `[B*N, W, D]`, что нормализовало математику внимания строго внутри границ каждого локального окна.

---

## 4. Математика Overlap-Add сборки окон

Простое суммирование перекрывающихся областей окон на выходе приводит к скачкообразному росту амплитуды сигнала в местах пересечений. Для восстановления исходного масштаба мы внедрили карту нормализации $M$:

$$y_{normalized}[t] = \frac{\sum_{n=1}^{N} y_n[t]}{M[t]}$$

Где $M[t]$ — количество окон, перекрывающих воксель времени $t$. В коде это реализовано эффективным накапливающим сложением:
```python
output[b, start_time:end_time, :] += window
norm_map[b, start_time:end_time, :] += 1.0  # Накапливаем веса наложений

norm_map = torch.clamp(norm_map, min=1.0)
output = output / norm_map  # Нормируем амплитуду
```
Это исключает искажение выходных масштабов и стабилизирует градиенты на обучении.

## 5. Обход Windows-специфичного бага KTM (WinError 6714)

На операционных системах Windows при импорте скомпилированной библиотеки `pyarrow` (зависимость Hugging Face `datasets`) ядро операционной системы часто блокирует дескрипторы папок проектов в кэше транзакций NTFS (TxF) [1.2.2]. 

Чтобы полностью обойти этот баг ядра Windows, мы исключили библиотеки `datasets`, `pandas` и `pyarrow` из зависимостей проекта (`pyproject.toml`). Вместо этого локальный загрузчик `CommonVoiceDataset` переписан на чистый встроенный Python-модуль `csv` для парсинга TSV-разметки. Это полностью убрало ошибку `WinError 6714` и сделало запуск проекта абсолютно стабильным.
